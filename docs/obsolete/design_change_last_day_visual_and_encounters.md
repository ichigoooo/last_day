# 设计变更记录：最后一天主视觉可选布局与遭遇随机性

- **状态**：提案 / 待架构评审  
- **日期**：2026-04-09  
- **背景**：综合版设计文档（`docs/人生最后一天_设计文档_综合版.md`）中「最后一天」强调自由行动、场景内叙事与资源约束；当前实现以选项推进为主，主视觉区长期为「左场景 + 右占位头像」，与「无对话对象时不应出现对象位」的交互预期不一致。  
- **目的**：把产品侧要求固化为可交接的变更说明，便于专业人员评估**界面结构、叙事协议与系统边界**的改法，而非在此文档中定稿具体技术方案。

---

## 1. 问题陈述

1. **随机性应在体验上可感知**：同一目的地（如「学校」）可能出现不同遭遇，选项与主视觉应随之变化，而不是固定为同一套版面。  
2. **主视觉区须随内容应变**：场景矢量图与对话对象头像均为 **optional**；仅当本帧叙事「真的有该内容」时才显示。  
3. **单元素居中**：若场景与对象**只存在其一**，该元素应在主视觉区内**居中**展示，另一侧不留空白占位。  
4. 本记录**不**替代综合版设计文档，仅补充**界面与遭遇层**的变更诉求。

---

## 2. 线框与布局规则（产品确认）

```
┌─────────────────────────────────────┐
│ 状态栏                               │
├─────────────────────────────────────┤
│  主视觉区（动态）                     │
│  ┌──────────┐ ┌──────────┐          │
│  │ 场景(可选) │ │ 对象(可选) │          │
│  └──────────┘ └──────────┘          │
│  二者皆有 → 并排                     │
│  仅其一   → 该项居中，不保留另一侧空位   │
├─────────────────────────────────────┤
│  旁白 / 选项区                       │
├─────────────────────────────────────┤
│  底栏（用户输入等）                   │
└─────────────────────────────────────┘
```

**硬性规则：**

| 场景图 | 对象图 | 预期布局 |
|--------|--------|----------|
| 有 | 有 | 两卡并排（比例可由设计再定） |
| 有 | 无 | 仅场景图，在主视觉区**水平居中** |
| 无 | 有 | 仅对象图，**水平居中** |
| 无 | 无 | 主视觉区收起或最小高度，避免「空框感」 |

## 3. 叙事场景示例（用于对齐「随机性」）

以下两例用于说明**同一地点、不同遭遇**下，选项与主视觉的差异；实现方式（纯 LLM / 规则表 / 混合）由架构评审决定。

### 3.1 遭遇 A：到场后无具体对话对象

- **情境**：玩家到达「学校」，叙事为在场内选择下一步去向。  
- **选项示例**：A 去教室 · B 去操场 · C 去老师办公室找以前的老师。  
- **主视觉**：仅有**场所**一侧有内容时，**不显示**对话对象头像（右侧不出现占位）。

### 3.2 遭遇 B：校门口与保安发生交互

- **情境**：在校门口被保安拦下。  
- **选项示例**：A 请求进入 · B 给老师打电话 · C 在校门口等待。  
- **主视觉**：除场所外，**显示对话对象**（保安），对象头像出现在主视觉区**对象位**；与遭遇 A 相比，右侧从「无」变为「有」。

## 4. 与当前实现的大致差距（截至记录时）

以下为便于评审对照的**观察性描述**，不排除后续代码已有局部演进：

- **最后一天界面**（`LastDayScreen` / `last_day.tscn`）长期呈现「左场所卡 + 右心境/占位图」的双卡结构，**未**按「无对象则隐藏对象位并单卡居中」做动态布局。  
- **叙事回合结构**（`NarrativeTurn` 等）以旁白 + 三选项 + 自定义提示为主；**是否**在结构化输出中携带「本帧是否展示场景/对象」「对象角色提示」等字段，需产品与技术共同定稿。  
- **随机遭遇**若完全依赖单次 LLM 输出，需考虑**离线兜底**与**内容安全**；若引入本地权重或遭遇表，需与现有 `LocationManager`、意图解析、`LastDayDirector` 的职责划分一并设计。

## 5. 建议架构评审时讨论的议题（非结论）

1. **叙事协议**：每回合 JSON（或等价结构）是否需显式字段，例如：`show_scene_image`、`show_character_frame`、`interlocutor_hint`（或枚举角色类型），以及默认值策略。  
2. **遭遇来源**：随机性来自模型一次生成、还是先由系统抽「遭遇模板」再调用模型填旁白与选项。  
3. **资源管线**：场景仍为 SVG/矢量卡；对象头像是占位图、统一立绘槽位，还是分角色资源表。  
4. **状态与回合**：地图传送、新回合开始、遭遇切换时，如何重置「对象是否显示」，避免残留上一帧对象头像。  
5. **可测性**：如何为布局规则与遭遇分支写最小可复现用例（含无 API 场景）。
