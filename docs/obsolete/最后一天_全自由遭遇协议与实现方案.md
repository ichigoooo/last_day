- 关联文档：

## 1. 文档目的

本文件用于把“人生最后一天真正实现框架中的全自由”落成一套**可执行的工程方案**。

本方案明确采用以下原则：

- 用户可以输入**任意目的地**与**任意行动**
- **遭遇什么、遇到谁、主视觉显示什么**，由 **LLM 决定**
- **时间、现金、电量、消息延迟、阶段推进、异常兜底**，由 **系统决定**
- 系统不预写固定遭遇表，不把“地点白名单”作为内容边界
- 系统只定义一套**严格的协议**，确保 LLM 生成的自由内容可以被解析、渲染、操作和回退

一句话：

**内容创作权交给 LLM，执行权和稳定性留给引擎。**

---

## 2. 什么叫“全自由”

本项目中，“全自由”不是“LLM 直接当 GM 改世界状态”，而是：

1. 用户可以自由说：
   - 我想去哪里
   - 我想见谁
   - 我想做什么
2. LLM 可以自由决定：
   - 这一回合是否真的抵达
   - 抵达后看到什么场景
   - 有没有具体人物
   - 这个人物是谁、什么身份、什么气质
   - 当前遭遇发生了什么
   - 给出哪三个选项
3. 系统只负责：
   - 将用户输入与系统资源状态送入协议
   - 校验 LLM 输出是否合格
   - 执行固定规则
   - 把内容渲染到 UI
   - 在输出非法或 API 不可用时兜底

因此，本方案中的“自由”是：

- **输入自由**
- **遭遇内容自由**
- **视觉内容自由**

但不是：

- 规则自由
- 资源自由
- JSON 结构自由
- 渲染行为自由

## 3. 设计决策

### 3.1 核心决策

项目采用“双协议、双调用、系统控规则”的运行时结构：

1. `Intent Parse`：LLM 负责理解用户这次在做什么
2. `System Resolve`：系统负责资源结算与硬规则推进
3. `Encounter Frame Render`：LLM 负责声明“这一帧世界如何呈现”
4. `Visual Render`：系统按协议渲染场景图、人物图、旁白与选项

### 3.2 明确不采用的方案

- 不采用“有限地点表决定所有遭遇”
- 不采用“系统预写学校/公司/公园若干固定事件”
- 不采用“LLM 直接改写 `WorldState`”
- 不采用“只改 UI 隐藏占位，不定义正式协议”
- 不采用“单次调用同时做意图解析、规则裁决、叙事生成、视觉字段生成、状态推进”的大一统 prompt

### 3.3 核心边界

LLM 可以决定：

- 叙事内容
- 遭遇类型
- 是否出现人物
- 人物的名字、身份、视觉描述
- 场景视觉描述
- 三个选项文案

LLM 不可以决定：

- 实际扣除多少时间
- 是否扣钱
- 是否扣电
- 消息何时回复
- 是否进入死亡阶段
- 上一帧状态是否保留
- 系统字段是否合法

## 4. 当前工程与目标之间的差距

当前实现已经具备以下基础：

- `TimeManager` 已实现真实时间推进到游戏时间
- `MoneySystem`、`BatterySystem`、`MessageSystem` 已可独立运行
- `LastDayDirector` 已有“意图解析 -> 系统结算 -> 叙事渲染”雏形
- `LastDay` 场景已有状态栏、主视觉、旁白区、选项区、手机和花钱叠层

但要实现本方案，还存在以下结构性缺口：

1. **主视觉不是协议驱动**
   - 当前为固定双卡布局
   - 无法根据“是否有人物/是否有场景”做四态切换

2. **回合模型过窄**
   - 当前 `NarrativeTurn` 只包含 `narration`、`options`、`custom_hint`
   - 不足以承载“地点呈现、人物呈现、遭遇声明、视觉 brief”

3. **没有正式的遭遇帧状态**
   - 当前 `WorldState` 没有“当前显示中的遭遇帧”
   - 无法保证切场、下一回合、地图传送时正确清理残留

4. **地点模型仍偏白名单**
   - 当前 `LocationManager` 主要服务于有限地点目录
   - 与“用户任意输入目的地”不匹配

5. **视觉资源管线只有场景卡**
   - 当前对象位仍是固定占位图
   - 没有统一的人物图生成与缓存协议

## 5. 目标架构总览

### 5.1 分层

运行时拆为五层：

1. **Input Layer**
   - 用户通过地图快捷入口或自然语言输入目的地/行动

2. **Interpret Layer**
   - LLM 输出 `IntentParseResult`
   - 系统只拿它做“理解参考”，不直接展示给用户

3. **Rule Layer**
   - 系统根据当前资源状态计算真实结算结果

4. **Encounter Layer**
   - LLM 输出 `EncounterFrame`
   - 这是当前回合真正的“世界显示协议”

5. **Render Layer**
   - Godot 根据 `EncounterFrame` 渲染 UI、主视觉、选项

### 5.2 数据流

```text
用户输入
  -> Intent Parse (LLM, JSON-only)
  -> System Resolve (C#)
  -> Encounter Frame Render (LLM, JSON-only)
  -> Validate / Sanitize / Default
  -> UI Render + Visual Asset Fetch
  -> 用户选择下一步
```

### 5.3 统一原则

- 任意地点输入，都先进入 `Intent Parse`
- 任意地点输入，不先问“在不在地点表中”
- 推荐地点只作为快捷入口，不作为内容边界
- 每一回合最终都要落成一份 `EncounterFrame`

## 6. 协议一：Intent Parse

`Intent Parse` 的目标不是写文案，而是帮助系统理解：

- 用户是不是想移动
- 这次行动大致属于什么类型
- 行动是否高风险
- 是否涉及手机/花钱/关系对象
- 行动大致耗时档位

### 6.1 输出对象：`IntentParseResult`

```json
{
  "version": 1,
  "destination_text": "南极",
  "is_travel_intent": true,
  "intent_type": "travel",
  "action_focus": "go_somewhere_far_away",
  "target_person": "",
  "cost_band": "medium",
  "screen_usage": "brief",
  "money_band": "none",
  "risk_level": "low",
  "wish_relevance": "high",
  "summary": "想离开这里，去一个极远的地方。"
}
```

### 6.2 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | int | 是 | 协议版本，首版固定 `1` |
| `destination_text` | string | 否 | 用户明确提到的目的地文本；没有则空字符串 |
| `is_travel_intent` | bool | 是 | 这次输入是否以“去某处”为主 |
| `intent_type` | enum | 是 | `travel` / `talk` / `message` / `post` / `buy` / `rest` / `look` / `remember` / `wait` / `other` |
| `action_focus` | string | 否 | 一句机器可读摘要，例如 `say_goodbye_to_someone` |
| `target_person` | string | 否 | 可能涉及的人物对象提示 |
| `cost_band` | enum | 是 | `instant` / `short` / `medium` / `long` / `xlong` |
| `screen_usage` | enum | 是 | `none` / `brief` / `active` |
| `money_band` | enum | 是 | `none` / `low` / `medium` / `high` |
| `risk_level` | enum | 是 | `low` / `medium` / `high` |
| `wish_relevance` | enum | 是 | `low` / `medium` / `high` |
| `summary` | string | 是 | 中文一句话概括 |

### 6.3 系统如何使用

- `is_travel_intent=true` 时，系统认为本回合可能涉及“地点变化”
- `cost_band`、`screen_usage`、`money_band` 只作为系统结算参考
- `summary` 进入日志和后续叙事 prompt
- `destination_text` 原样保留，供 `EncounterFrame` 阶段理解“用户实际想去哪里”

### 6.4 约束

- `Intent Parse` 不能直接输出 UI 字段
- 不能输出 `show_scene_image`、`show_character_frame`
- 不能输出 SVG
- 不能输出选项
- 不能直接决定真实耗时分钟数

## 7. 协议二：Encounter Frame

`EncounterFrame` 是本方案的核心协议。

它定义了**这一回合系统到底应该显示什么**。

### 7.1 输出对象：`EncounterFrame`

```json
{
  "version": 1,
  "frame_id": "enc_001",
  "place_name": "南极补给站外的雪地",
  "place_subtitle": "风雪比想象中更近",
  "arrival_mode": "symbolic_arrival",
  "encounter_kind": "obstruction",
  "encounter_summary": "你刚靠近门口，就先被拦了下来。",
  "show_scene_image": true,
  "scene_visual_brief": "大片风雪中的空旷白地，远处是低矮补给站轮廓",
  "show_character_frame": true,
  "character_name": "值班员",
  "character_role": "service_worker",
  "character_visual_brief": "厚重防寒服，护目镜遮住大半张脸",
  "narration": "风把话吹散了一半。你刚靠近门口，就有人抬手示意你停下。",
  "options": [
    { "id": "a", "label": "说明你为什么来这里", "action_hint": "talk" },
    { "id": "b", "label": "请求借一个能联系外界的设备", "action_hint": "ask" },
    { "id": "c", "label": "先站在风里看一会儿", "action_hint": "wait" }
  ],
  "custom_hint": "也可以自己输入此刻真正想做的事。"
}
```

### 7.2 顶层字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | int | 是 | 固定 `1` |
| `frame_id` | string | 否 | LLM 可生成，系统可忽略或重写 |
| `place_name` | string | 是 | 当前回合向用户展示的地点名称 |
| `place_subtitle` | string | 否 | 地点副标题，可为空 |
| `arrival_mode` | enum | 是 | 见 7.3 |
| `encounter_kind` | enum | 是 | 见 7.4 |
| `encounter_summary` | string | 是 | 当前遭遇一句话概括 |
| `show_scene_image` | bool | 是 | 是否显示场景图 |
| `scene_visual_brief` | string | 条件必填 | `show_scene_image=true` 时必须有值 |
| `show_character_frame` | bool | 是 | 是否显示人物图 |
| `character_name` | string | 条件必填 | `show_character_frame=true` 时建议有值 |
| `character_role` | enum | 条件必填 | `show_character_frame=true` 时必须合法 |
| `character_visual_brief` | string | 条件必填 | `show_character_frame=true` 时必须有值 |
| `narration` | string | 是 | 当前旁白 |
| `options` | array | 是 | 最多 3 条，最少 1 条，系统最终保底补齐到 3 条 |
| `custom_hint` | string | 是 | 自定义输入提示 |

### 7.3 `arrival_mode` 枚举

| 值 | 含义 |
|----|------|
| `stay_here` | 用户没有真正移动，仍在当前地点 |
| `physical_arrival` | 叙事上视为真实到达 |
| `symbolic_arrival` | 叙事上以象征或压缩方式到达 |
| `memory_projection` | 进入记忆/投射/想象型场景 |
| `blocked_arrival` | 用户试图到达，但停在边界或门口 |
| `not_arrivable_today` | 今天不能真正抵达，叙事转为“无法抵达” |

说明：

- 该字段服务于叙事解释与 UI 呈现，不直接决定时间规则
- “去南极”“回小时候的家”“去一个不存在的地方”都可以通过该字段合法表达

### 7.4 `encounter_kind` 枚举

| 值 | 含义 |
|----|------|
| `solitude` | 只有环境，没有具体人物 |
| `conversation` | 有对话对象 |
| `obstruction` | 有拦阻、门槛、边界、审视 |
| `search` | 用户在寻找某人或某物 |
| `missed_connection` | 来晚了、错过了、人不在 |
| `waiting` | 停顿、等待、迟疑 |
| `memory_echo` | 回忆/投射/象征性遭遇 |
| `other` | 其他，留给模型自由发挥 |

说明：

- 该字段不是在写死剧情，只是在给系统提供一个**机器可读的遭遇类型**
- 不同地点可以返回同一个 `encounter_kind`
- 同一个地点每次都可以返回不同 `encounter_kind`

### 7.5 `character_role` 枚举

| 值 | 含义 |
|----|------|
| `authority` | 有制度权力的人 |
| `service_worker` | 工作人员、前台、值班员等 |
| `teacher` | 老师、导师、前辈 |
| `coworker` | 同事、上级、下属 |
| `friend` | 朋友 |
| `family` | 家人 |
| `lover` | 恋人、前任、暧昧对象 |
| `stranger` | 陌生人 |
| `self` | 自己的投射、自我分身 |
| `reaper` | 死神 |
| `other` | 其他 |

说明：

- `character_name` 可以自由
- `character_role` 必须有限枚举，便于未来做风格化视觉和日志分析

### 7.6 `options` 子对象

```json
{
  "id": "a",
  "label": "说明你为什么来这里",
  "action_hint": "talk"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | `a` / `b` / `c` 或其他短 id |
| `label` | string | 是 | 给用户展示的选项文案 |
| `action_hint` | string | 否 | 给系统或日志使用的短标签 |

约束：

- `label` 必须是可执行动作
- 优先 6 到 18 个汉字
- 最多 3 条
- 少于 3 条时系统补齐兜底项

## 8. Encounter Frame 的系统解释规则

这部分决定“LLM 可以自由写，但系统怎样稳定吃下去”。

### 8.1 主视觉解释

| `show_scene_image` | `show_character_frame` | 系统行为 |
|--------------------|------------------------|----------|
| `true` | `true` | 双卡并排 |
| `true` | `false` | 仅场景卡居中 |
| `false` | `true` | 仅人物卡居中 |
| `false` | `false` | 主视觉区收起或降到最小高度 |

### 8.2 字段冲突时的行为

1. `show_scene_image=false` 且 `scene_visual_brief` 非空
   - 系统忽略 `scene_visual_brief`

2. `show_character_frame=false` 但带了 `character_name`
   - 系统忽略人物字段

3. `show_character_frame=true` 但人物字段不合法
   - 系统把 `show_character_frame` 强制降为 `false`

4. `options` 为空
   - 系统补 3 个兜底项

5. `narration` 为空
   - 系统使用默认旁白

### 8.3 帧切换规则

应用新 `EncounterFrame` 前，系统必须：

1. 清空上一帧人物显示状态
2. 清空上一帧场景副标题和人物名
3. 取消上一帧仍在加载中的人物图请求
4. 重置主视觉容器尺寸策略
5. 再按新帧重新装配

这条规则是防止“上一帧保安残留到下一帧海边”这类错误的关键。

## 9. 视觉资源协议

本方案要求场景图和人物图都走统一描述式生成。

### 9.1 原则

- 系统不要求 LLM 返回 SVG 原文
- 系统要求 LLM 返回**视觉 brief**
- 真实 SVG 生成由专门的 visual prompt 完成

### 9.2 两类视觉 brief

1. 场景 brief
   - 来源：`scene_visual_brief`
   - 输入到通用 SVG 生成服务

2. 人物 brief
   - 来源：`character_visual_brief`
   - 输入到同一 SVG 生成服务，但 `subject_kind=character`

### 9.3 缓存策略

缓存键不再只用 `locationId`，而改为：

- `scene::<hash(scene_visual_brief)>`
- `character::<role>::<hash(character_visual_brief)>`

这样“同一地点不同遭遇”才不会复用错误场景图。

### 9.4 失败兜底

- 场景 SVG 失败：显示统一极简场景底图
- 人物 SVG 失败：显示统一人物轮廓占位
- 任一生成失败不阻断回合流程

## 10. 系统状态模型改造

### 10.1 `WorldState` 建议新增字段

| 字段 | 说明 |
|------|------|
| `CurrentPresentedPlaceName` | 当前 UI 展示的地点名 |
| `CurrentEncounterFrameJson` | 当前帧的原始 JSON 或序列化结果 |
| `CurrentArrivalMode` | 当前到达模式 |
| `CurrentEncounterKind` | 当前遭遇类型 |
| `CurrentCharacterName` | 当前人物名 |
| `CurrentCharacterRole` | 当前人物角色 |
| `LastResolvedDestinationText` | 上一次用户提到的目的地原文 |
| `VisualAssetCache` | brief-hash 到 SVG/纹理缓存 |

### 10.2 哪些状态不应进入长期存档

以下只建议作为运行时缓存，不要求写入可恢复存档：

- 当前场景图 SVG 原文
- 当前人物图 SVG 原文
- 正在请求中的视觉任务
- 上一帧纹理引用

### 10.3 状态重置时机

以下时机必须清当前帧人物态：

- 进入 `LastDay` 新局
- 地图传送完成
- 自由输入触发新地点叙事
- `EncounterFrame` 应用前
- 进入死亡阶段

## 11. 模块职责重划分

### 11.1 `LocationManager`

从“有限地点权威目录”改为：

- 推荐地点快捷入口
- 本地别名与展示名参考
- 无 API 时的保守兜底

不再负责：

- 决定用户能不能去哪里
- 决定某地一定会发生什么

### 11.2 `LastDayDirector`

成为唯一编排入口，职责变为：

1. 接收用户输入
2. 调 `Intent Parse`
3. 结算系统资源
4. 组装 `EncounterFrame` prompt
5. 解析、校验、清洗 `EncounterFrame`
6. 写入 `WorldState`
7. 返回 UI 层展示

### 11.3 `VisualSvgService`

建议从当前 `LocationCardSvgService` 扩成通用视觉服务：

- `GetSceneTextureAsync(sceneVisualBrief)`
- `GetCharacterTextureAsync(role, characterVisualBrief)`

### 11.4 `LastDayScreen`

UI 层只做三件事：

- 绑定状态栏与输入事件
- 应用 `EncounterFrame`
- 管理主视觉布局与异步纹理加载

UI 层不做：

- 规则裁决
- 地点语义理解
- 文案拼接

## 12. Prompt 设计原则

### 12.1 `Intent Parse` Prompt

目标：

- 把输入压缩为机器可执行标签
- 输出短小稳定 JSON

禁止：

- 直接写叙事文案
- 输出人物和场景显示字段

### 12.2 `Encounter Frame Render` Prompt

目标：

- 按系统已经结算好的结果，声明当前一帧世界

必须要求模型：

- 只输出 JSON
- 显式输出 `show_scene_image`
- 显式输出 `show_character_frame`
- 不依赖上一帧
- 不修改系统已结算资源

### 12.3 视觉 Prompt

## 13. 异常处理与兜底

### 13.1 LLM 无响应

回退到默认 `EncounterFrame`：

- 地点名：当前输入原文或“此刻所在之处”
- 无人物
- 一段保守旁白
- 3 个兜底选项

### 13.2 JSON 非法

策略：

1. 尝试解析
2. 失败则写日志
3. 回退默认帧

不做：

- 把原始非法文本直接展示给用户

### 13.3 字段部分缺失

策略：

- 用字段级默认值修复
- 不因局部错误放弃整个回合

### 13.4 SVG 生成失败

策略：

- 立刻替换成本地极简占位
- 后台日志记录
- 不打断用户当前选择

### 13.5 内容安全

继续保留现有：

- `CrisisKeywordGuard`
- `ContentSafetyFilter`

同时增加：

- `EncounterFrame` 字段级过滤
- `character_name`、`place_name`、`narration`、`label` 的显示层清洗

## 17. 验收标准

当以下条件全部成立时，可认为“全自由协议版 LastDay”达标：

1. 用户输入任意地点，不会先被地点表拒绝
2. LLM 可以在不同回合自由决定不同地点呈现与不同人物
3. 系统可以稳定解析合法 `EncounterFrame`
4. 非法 `EncounterFrame` 不会打断流程
5. 主视觉四态全部正确
6. 上一帧人物不会残留到下一帧
7. 场景图和人物图都能按 brief 生成或按失败策略回退
8. 时间、现金、电量、消息延迟仍由系统稳定控制
9. 无 API 时仍可进入保守兜底体验
10. 现有“最后一天”四机制不被破坏

---

## 18. 结论

本方案的核心不是“让 LLM 无限发挥”，而是：

- 允许 LLM 在**内容层**无限发挥
- 用协议把这些内容限制在**系统可以执行的格式**里

因此，本方案既保留了“人生最后一天”最重要的自由感，也保住了 Godot 工程落地所需的稳定性、可测性和可维护性。

最终结论可概括为一句话：

**不再预写遭遇表，不再把地点表当边界；改为让 LLM 直接声明每回合世界，而系统只负责解释这份声明。**
