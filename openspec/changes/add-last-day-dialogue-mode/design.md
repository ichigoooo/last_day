## Context

当前 `LastDay` 阶段已经有较完整的规则层与场景渲染链：`LastDayDirector` 负责输入解析与系统结算，`EncounterFrame` 负责地点/人物/旁白/选项呈现，`LastDayScreen` 负责把回合结果映射到主视觉、旁白区和底部输入区。这套结构足以支持“行动 -> 遭遇 -> 旁白”的单回合体验，但它把“人物出现”定义成了场景中的一个描述元素，而不是一个可持续交互的对象。

这导致三个直接问题：

- 当场景中出现人物时，内容仍然以第三人称旁观式旁白为主，无法兑现“当面对话”的目标体验。
- UI 只有一套“行动输入”语义，没有“正在和某人说话”的独立模式，因此人物头像、文本区、按钮和输入框都无法表达当前是会话而不是行动。
- 活动日志只能记录“输入 + 旁白 + 意图摘要”，无法保留说话人、回应轮次和结束对话这类对终局总结真正重要的信息。

本次变更是一次跨模型、导演流程、提示词、UI 状态和日志结构的架构升级，但它仍需保留已有约束：

- 人物、人物发言、回应选项和现场对话内容都允许由 LLM 生成，不依赖预置人物表。
- 时间、现金、电量、消息、内容安全、异常兜底和阶段推进继续由系统掌控。
- 对话模式中必须禁用“前往场所”“手机”“死神花钱”等会破坏当前会话语义的入口。
- 第一版不要求建立跨整局稳定的人物库；重点是把“遇见某人 -> 进入会话 -> 记录会话 -> 回到行动”做通。

## Goals / Non-Goals

**Goals:**

- 让“最后一天”在遇到可交互人物时进入面对面对话模式，而不是继续使用环境旁白代替对话。
- 允许 LLM 生成现场人物身份、开场对白、玩家回应选项、自定义回应提示和继续/结束对话结果。
- 为运行时增加明确的对话会话状态，使系统能够区分“行动模式”和“对话模式”。
- 让 `LastDay` UI 在两种模式间切换，并在对话模式中高亮对象、显示说话内容、重映射输入语义、禁用冲突入口。
- 将面对面对话以结构化方式写入活动日志、近期故事压缩输入和终局总结输入。
- 保留现有 `EncounterFrame`、主视觉资源链和系统结算边界，尽量以增量方式完成迁移。

**Non-Goals:**

- 不在本次变更中引入预置 NPC 数据库、关系数值系统或跨整局人物持久记忆。
- 不把手机聊天、朋友圈或死神花钱统一进同一会话系统。
- 不要求这次变更重做 `LastDay` 之外的其他阶段。
- 不要求为对话模式一次性完成所有视觉打磨、动画和美术特效。

## Decisions

### Decision 1：引入显式的运行时对话会话，而不是把一切继续塞进 `EncounterFrame`

新增两类运行时模型：

- `DialogueSession`：记录当前对话对象、地点、会话摘要、最近轮次、会话是否仍在进行
- `DialogueTurn`：记录当前回合的说话者、舞台说明、当前说话内容、玩家回应选项、自定义提示、是否允许结束

`EncounterFrame` 继续负责场景事实和视觉呈现，但不再独自承担“持续对话状态”的职责。其责任变为：

- 描述玩家当前来到哪里、看见谁、此刻是 ambient 还是 dialogue
- 在首次进入对话时携带开场用的 `DialogueTurn`

之所以不继续扩充 `EncounterFrame` 去承载完整多轮会话，是因为：

- 场景帧是“这一刻看见什么”，会话状态是“这段关系正在进行什么”
- 多轮会话需要保留前后文与结束条件，这类状态不适合做成每回合一次性的展示对象
- 单独的 `DialogueSession` 更利于日志、恢复、重置和 UI 模式切换

替代方案：

- 把所有对话字段直接塞进 `EncounterFrame`：实现看似更快，但会把场景快照和会话状态长期耦合
- 完全独立出一个新的对话场景：边界清楚，但会破坏现有 `LastDay` 的连续体验和系统共享

### Decision 2：首次遇人由遭遇结果直接携带“开场对白”，后续续聊走专用对话提示词

导演流程调整为：

1. 玩家在行动模式输入一句行动
2. 系统完成意图解析与资源结算
3. 遭遇渲染阶段生成 `EncounterFrame`
4. 若本回合进入对话，`EncounterFrame` 同时携带开场 `DialogueTurn`
5. 系统创建 `ActiveDialogueSession`
6. 后续玩家输入不再走行动解析，而是直接推进当前对话

选择“首次进入对话时一次返回场景 + 开场对白”，而不是“先返回场景再单独请求第一句对话”，原因是：

- 可以减少首次入对话时的额外等待
- 场景与第一句对白通常共享同一上下文，一次生成更容易保持语气一致
- 只把额外模型调用留给真正的“继续对话”，不会让每次普通遇人都变慢

替代方案：

- 首次进入对话时再打第三次调用生成开场对白：更模块化，但会进一步拉长交互延迟
- 永远使用同一个 prompt 处理 ambient 与 dialogue：契约会变得过大，JSON 稳定性和调试性都会下降

### Decision 3：活动对话期间绕过行动解析，使用专用的对话推进路径

当 `WorldState.ActiveDialogueSession` 存在时：

- 底部输入统一解释为“你怎么回应”
- 主选项统一解释为“玩家回话”
- `LastDayDirector` 不再尝试把输入解析成旅行、消费、发消息或其他行动
- 玩家若想结束当前会话，只能通过明确的“结束对话”入口

系统仍会对每轮对话施加小幅、固定且由系统定义的时间推进，不把资源变化决定权交给 LLM。电量、金钱和消息系统不由面对面对话直接驱动。

这样做的原因是：

- 对话中的任意输入都天然更像语用回应，而不是世界行动
- 若继续共用行动解析，模型很容易把“我先走了”“你别说了”误判成地点移动或其他系统动作
- 固定的系统侧对话耗时更能防止 LLM 越权决定对话代价

替代方案：

- 对话期间继续走 `IntentParseResult`：复用度高，但语义误判风险很大
- 让 LLM 决定每轮对话耗时：更灵活，但会打破系统权威边界

### Decision 4：`LastDay` 维持单场景，但新增清晰的 UI 模式切换规则

`LastDayScreen` 不拆新场景，而是在现有结构内支持两种展示模式：

- `Ambient`：延续当前的行动/遭遇视图
- `Dialogue`：显示高亮对象、当前说话内容、玩家回应选项、自定义回应输入和“结束对话”按钮

进入 `Dialogue` 模式时，UI 必须：

- 高亮人物头像容器和对象名称
- 将文本主区域解释为“当前这人说的话”
- 将三颗主选项解释为“你可以怎么回”
- 将输入框 placeholder 和提交按钮文案改成回应语义
- 禁用地图、手机、死神花钱等入口

退出 `Dialogue` 模式时，UI 必须恢复行动语义并重新启用这些入口。

选择保留单场景的原因是：

- 状态栏、主视觉、叙事容器和底部工具本来就共享同一组系统状态
- 用户不应感受到“切到另一个场景”，而应感受到“此刻正在和人说话”
- 对现有 Godot 场景结构的破坏更小

替代方案：

- 单独创建一个对话场景或 overlay：边界更硬，但状态同步和切换成本更高

### Decision 5：对话历史以结构化活动记录为准，不再压扁成普通旁白

`SessionActivityLog` 新增面对面对话相关记录类型，至少需要保留：

- 会话 id
- 地点
- 说话者角色（player / npc / system）
- 说话者名称
- 文本内容
- 事件类型（开场、继续、结束）

`WorldState.RecentTurnLines` 与终局提示词上下文也要纳入这些记录，但不再只写成“输入：… 旁白：…”这种压缩串。压缩时可以保留摘要，但原始活动日志必须保留说话归属。

这样做的原因是：

- 终局总结真正需要的是“你和谁说了什么”，不是“旁白概述你们说了什么”
- 后续若要做人物连续性、悼词引用或关系回收，结构化日志比大段自由文本可用得多

替代方案：

- 继续把对话合并进 `last_day_turn` 文本：实现简单，但后续总结和检索会失真

### Decision 6：LLM 负责生成人物与对白，系统负责约束和兜底

本次变更中，LLM 可以生成：

- 场景中遇到的人是谁
- 这个人现在对你说什么
- 你可以如何回应
- 对话继续后的语气、走向和结束时机

系统负责：

- 是否进入或退出对话模式
- 对话期间的 UI 锁定规则
- 内容安全过滤
- 结构化 JSON 解析与字段补默认值
- 资源结算和阶段推进
- API 不可用或返回非法时的回退显示

这样既满足“人物与对白都可由 LLM 生成”，也避免把系统控制权交给模型。

替代方案：

- 预置角色池 + 模板对白：更稳，但和用户已确认的目标冲突
- 让 LLM 同时决定模式、资源与阶段：更自由，但风险不可控

## Risks / Trade-offs

- [Risk] 多轮会话会引入新的运行时状态，若清理不当，可能在对话结束后残留禁用按钮、旧人物高亮或旧选项。
  → Mitigation：把 `ActiveDialogueSession` 作为单一事实来源，并在进入/退出会话时集中执行 UI 重置。

- [Risk] LLM 生成的人物可能在同一会话中发生名称、语气或关系漂移。
  → Mitigation：创建 `DialogueSession` 时冻结当前对象身份与话题摘要，续聊 prompt 只能在该快照上展开。

- [Risk] 新增对话 prompt 会增加实现复杂度和回合等待时间。
  → Mitigation：让开场对白并入遭遇结果，只把专用调用留给续聊回合。

- [Risk] 若对话可以无限继续，玩家可能长时间停留在会话中，影响整体节奏。
  → Mitigation：始终提供“结束对话”入口，并让每轮对话消耗系统定义的小幅时间。

- [Risk] 把所有人物交给 LLM 生成后，不同地点可能出现过于漂浮或不够可信的人物配置。
  → Mitigation：续用地点、当前状态、人物关系线索和近期故事摘要作为 prompt 约束，并在系统侧为空字段和离谱输出提供降级策略。

- [Trade-off] 单场景多模式切换比新场景切换更省改动，但 `LastDayScreen` 责任会进一步增大。
  → 回报是当前系统状态和视觉布局可以被连续复用，玩家体验更完整。

## Migration Plan

1. 新增运行时模型与提示词契约
   - 增加 `DialogueSession`、`DialogueTurn` 及相关 JSON 解析/默认值逻辑
   - 新增或改造遭遇渲染 prompt，使其可以声明 ambient/dialogue 并在需要时返回开场对白
   - 新增对话续聊 prompt

2. 扩展 `WorldState` 和 `LastDayTurnResult`
   - 增加当前活动会话、当前展示模式、对话缓存等字段
   - 保留旧 `NarrativeTurn` 兼容层，确保 UI 可分步迁移

3. 重构 `LastDayDirector`
   - 增加“若当前有活动会话则继续对话，否则按行动回合处理”的双路径
   - 在进入、继续、结束对话时统一写活动日志与近期摘要输入

4. 更新 `LastDay` UI 和场景
   - 加入对话模式专用的按钮状态、输入语义、对象高亮和结束按钮
   - 验证对话中禁用地图、手机、死神花钱，退出后恢复

5. 接入总结链路
   - 更新 `SessionActivityLog`、`story_summary` 上下文和终局输入组装逻辑，使面对面对话可被回收

6. 回滚策略
   - 若新会话链路不稳定，可暂时让遭遇结果统一退回 ambient 模式
   - UI 保留旧行动流分支，忽略 `ActiveDialogueSession` 即可恢复当前体验

## Open Questions

- 第一版是否需要把“舞台说明”单独做成一行淡色文本，还是先与说话内容共用现有叙事容器。
- 后续版本是否需要让同一人物在分离的两次遭遇之间保持更强的一致身份记忆；本次变更暂不阻塞于此。
