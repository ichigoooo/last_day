## Context

项目现在最强的部分，是中段的“最后一天”自由行动系统；最弱的部分，是首尾。

已有实现已经证明：

- `LastDay` 的地点、资源、消息、现场对话系统足以支撑一整段高自由度旅程。
- `SessionActivityLog`、`ClosurePromptVars` 与终局提示词已经具备把玩家行为回收到收束文本里的基础能力。
- 仓库中已安装 `Dialogic` 插件，足以承载固定对白、选择、文本输入、变量与桥接调用。

但当前旅程仍存在结构性问题：

- 序章以表单为核心，叙事被交互形式稀释。
- `FinalWish` 过早出现，造成中段被“任务化”。
- 结尾被拆成多个步骤，玩家在“点下一步”，而不是在被一条完整情绪线带走。
- 高自由中段与固定首尾之间缺乏明确的叙事协议，导致体验像“程序 + 程序 + 程序”的串联。

因此，本次设计的重点不是增加更多系统，而是重新安排“哪些部分应该固定，哪些部分应该自由；哪些内容应该先发生，哪些内容必须后置”。

## Design Thesis

这次重构遵循一个核心命题：

> 先让玩家完整经历一次像真的一样的失去，再把时间还给他。

这意味着：

- 开头必须足够精雕细琢，让玩家不是在“输入资料”，而是在“被归档”。
- 中段仍然保持不被束缚，因为真正有力量的不是被编好的选项，而是玩家自己的犹豫、拖延、删改与绕行。
- 终章必须先完成“死亡仪式”，包括遗愿、讣告、墓志铭与静默，然后才告诉玩家：你没有死。
- 真相揭示不能像反转笑话，而应像一只手在最后把人轻轻送回现实。

## Goals / Non-Goals

**Goals**

- 让开头与结尾达到视觉小说级别的文案完成度与节奏控制。
- 使用 Dialogic 承载固定叙事节点，而不是继续把关键节点写成散落在各界面的功能逻辑。
- 将 `工作 / 关系 / 逃避` 的信息采集自然融入死神对话。
- 将 `FinalWish` 后置到终章弥留时刻。
- 保留 `讣告`、`墓志铭` 与分享卡，但使它们成为终章仪式的一部分。
- 保留 `LastDay` 的高自由度和面对面对话系统。
- 让“你没有死”的真相在体验尾声产生升华，而不是削弱前面的重量。

**Non-Goals**

- 不把整个游戏改造成传统分支树视觉小说。
- 不推翻现有 `LastDay` 的核心资源系统与自由探索结构。
- 不在本次设计中引入复杂的多结局树或大规模预置 NPC 数据库。
- 不要求所有书写都改成 LLM 实时生成；关键节点优先采用固定文案与少量动态插槽。

## Core Decisions

### Decision 1：Dialogic 只接管“固定首尾”，不接管整个中段

`Dialogic` 的最大价值，不在于把整个游戏都变成对白树，而在于让必须精雕细琢的段落拥有可控的节奏、停顿、镜头切换和交互语义。

因此：

- 序章与终章使用 `Dialogic`。
- `LastDay` 自由探索继续由现有 C# 运行时驱动。
- `Dialogic` 通过变量与桥接调用读写 `GameSession`、`ActivityLog` 和终局生成结果。

这样做的原因：

- 中段的独特性来自高自由度，不应被过度脚本化。
- 首尾则恰恰需要固定文案和可控节奏。
- 只让 `Dialogic` 接管首尾，能最大化收益，同时最小化对现有系统的破坏。

### Decision 2：开头改成“档案处对话”，不再保留三段表单式登记

当前 `DeathRegistration` 的三块文本输入，本质上是在收三类信息：

- 你如何定义自己的工作与生存状态
- 你最牵挂的人际关系
- 你在逃避什么

这些信息仍然要保留，但收集方式必须改变。

新的开头改为：

- 黑屏系统通知先建立“你已死亡”的事实。
- 随后进入死神档案处。
- 死神用三轮短问答，分别套取 `work / relation / escape`。

推荐问法不是功能标题，而是人物对白：

- “档案第一页总得写点体面的东西。你平时怎么向别人介绍自己？”
- “联系人那一栏，我只抄第一个浮上来的名字。你想到谁了？”
- “未处理事项很多人会写假话。你一直在躲什么？”

每轮之后，死神给一句极短回声，不分析，不鸡汤，只像档案边角批注。

这样做的原因：

- 保留结构化输入，却消除“在填问卷”的感觉。
- 让用户从第一分钟起就进入戏剧语境。
- 为死神建立清晰的人物气质：冷静、办事、稍刻薄，但不恶意。

### Decision 3：`FinalWish` 必须后置到终章弥留时刻

遗愿是一句在时间真正见底时才有分量的话。它不该出现在玩家还没真正经历这一天之前。

因此：

- `Verdict` 不再要求填写遗愿。
- 中段不再以遗愿作为显性任务目标。
- `FinalWish` 只在终章、强制接管后登记。

新的台词语义应该是：

- “遗愿栏先空着。”
- “活人还没疼的时候，最会说漂亮话。”
- “等你真的只剩一点电，再填。”

这样做的原因：

- 避免中段被游戏化为“完成遗愿任务”。
- 让终章的输入成为被逼出来的真话，而不是早早写下的计划。
- 让“遗愿”与“回到现实的一句”形成两层不同重量的回答。

### Decision 4：保留讣告与墓志铭，但放在真相揭示之前

这是本次设计最重要的结构判断之一。

终章顺序必须是：

1. 玩家以为自己真的死去
2. 系统与死神完成归档
3. 讣告成立
4. 墓志铭成立
5. 真相才被揭示

而不是先告诉玩家“其实你没死”，再展示讣告与墓志铭。

原因：

- 先成立死亡仪式，玩家才会短暂地接受失去。
- 真相后揭示，主题才会从“死亡模拟”升华到“重新拥有时间”。
- 如果先揭示真相，讣告和墓志铭会立刻失去重量，沦为道具。

### Decision 5：删除独立的“遗言一句”步骤

现有 `Funeral` 里的“遗言一句”会与以下内容重复：

- 终章的 `FinalWish`
- 终章之后的“回到现实的一句”
- 墓志铭中可分享的一句话功能

因此：

- 移除独立的“遗言”步骤。
- 其“留下最短一句可被记住的话”的功能，由 `FinalWish` 与 `墓志铭` 分担。
- 其“回到现实的个人承诺”功能，由真相揭示后的最后一个问题承担。

这样做的原因：

- 减少终章中的概念竞争。
- 让每一句重要文本都有明确职责。
- 缩短仪式步骤，避免再次形成 wizard 式体验。

### Decision 6：中段仍需叙事锚点，但不再依赖 `FinalWish`

遗愿后置后，中段提示词与故事压缩链路不能再使用 `final_wish` 作为叙事锚。

新的中段锚点建议定义为：

- `ArchiveSummary`：档案处对话后由系统或 LLM 提炼出的用户惯性摘要
- `DanglingThreads`：当前最未了的一两条线索，如“没拨出去的电话”“不愿回去的家”“一直绕开的地点”

它们的作用不是给玩家下任务，而是给 LLM 一个低侵入的叙事重心。

这样做的原因：

- 中段依旧能生成更聚焦的遭遇和对白。
- 但玩家不会被显性任务牵着走。
- 旅程保留“像生活而不是像关卡”的气质。

### Decision 7：关键节点优先采用固定文案，动态生成只填受控插槽

本次重构要的是完成度，不是更高的不确定性。

因此：

- 黑屏通知、档案处主干对白、23:50 接管、真相揭示等关键节点，优先使用固定文案。
- 动态生成只用于：
  - 死因
  - 档案批注
  - 中段遭遇与面对面对话
  - 终章遗愿候选
  - 讣告
  - 墓志铭候选
  - 最终系统报告中的统计数据

这样做的原因：

- 首尾的语气必须被雕出来，不能全部交给即时生成。
- 固定骨架 + 动态插槽，更容易保持一致的美学控制。
- API 失败时也能优雅降级，不至于首尾失真。

### Decision 8：将不同“声音”明确分层

本次旅程需要至少三种不同声音：

| 声音 | 功能 | 气质 |
|------|------|------|
| 系统 | 宣告事实、展示报告、夺回控制权 | 冷、准、无情绪 |
| 死神档案员 | 归档、提问、登记遗愿、提供讣告语境 | 克制、荒诞、稍有锋利 |
| 回到现实的旁白 | 揭示真相、把人送回现实 | 安静、温和、不过分安慰 |

这三种声音不能混成一种“作者口吻”，否则会削弱层次。

## Stage-by-Stage Design

### Stage A：黑屏通知

**承载目标**

- 立刻建立“死亡已发生”的事实。
- 让用户被动接受设定，而不是通过死神解释设定。
- 引入系统 OS 隐喻。

**交互形式**

- 纯黑背景
- 白色等宽字
- 顶部高优先级通知
- 只有一个按钮：`确认已知晓`

**文案原则**

- 使用 [docs/叙事优化.md] 中“冷静系统通知”的基调。
- 不要一开始就抒情。
- 不用解释规则的世界观背景，只陈列结果。

**工程归属**

- Dialogic 时间线 `timeline_intro_boot`
- 主要为固定文案

### Stage B：档案处对话

**承载目标**

- 将 `work / relation / escape` 融入人物对白。
- 建立死神的气质。
- 让玩家感到“被归档”，而不是“被评估”。

**交互形式**

- Dialogic 角色对话
- 每轮一个短文本输入
- 每轮一个短回声
- 末尾出现档案批注

**必须保留的数据映射**

| 对话轮次 | 存储字段 |
|----------|----------|
| 自我介绍 / 生存状态 | `Soul.WorkText` |
| 第一个想到的人 | `Soul.RelationText` |
| 一直在躲什么 | `Soul.EscapeText` |

**文案原则**

- 每个问题都要像一句活对白，而不是字段名。
- 每个回声不超过 1 到 2 句。
- 不做心理诊断，不替玩家总结人生。

**删改要求**

- 删除当前三大表单标题式体验。
- 保留现有标签提取与死因生成链路，只更换输入入口。

### Stage C：死因裁定与放行

**承载目标**

- 从“档案处”平滑过渡到“最后一天”。
- 宣布死因，但不把遗愿放在这里。

**交互形式**

- 一页极短过渡
- 展示死因
- 死神开场白仅作放行，不要求玩家长输入

**文案原则**

- 死因是系统裁定
- 死神只负责轻轻把人推入最后一天
- 必须明确埋下“遗愿稍后再填”

**删改要求**

- 删除当前 `Verdict` 的遗愿输入框和建议按钮
- 保留死因与开场白展示价值

### Stage D：最后一天自由探索

**承载目标**

- 保留项目现有最有生命力的部分
- 让玩家自己产出情绪与行为痕迹

**交互形式**

- 维持现有 `LastDay`
- 地图、手机、死神花钱、现场对话继续存在

**设计约束**

- 不新增大量强制剧情节点
- 只在接近终点时强制介入
- 中段提示词改用 `ArchiveSummary / DanglingThreads`

**删改要求**

- 中段不再以“完成遗愿”为语义中心

### Stage E：23:50 系统强制接管

**承载目标**

- 让失去控制权本身成为叙事
- 把自由行动收束为终章仪式

**交互形式**

- 屏蔽地图、手机、行动输入
- 时钟放大
- UI 响应变迟缓或失效
- 对话与环境音逐步剥离

**文案原则**

- 先让系统收权，再让死神出现
- 不要一下子倾倒长篇文字
- 玩家应先“感觉到不能再行动”，再“读到相关文案”

**工程归属**

- `LastDay` 末端触发
- Dialogic 时间线 `timeline_closure_lockdown`

### Stage F：弥留遗愿

**承载目标**

- 让 `FinalWish` 成为一句被逼出来的真话

**交互形式**

- Dialogic 文本输入
- 可选提供 2 到 3 个候选
- 允许自填

**推荐语义**

- “还剩这一点电。你想留给谁？”
- “这次我替你照原话登记，不帮你润色。”

**设计约束**

- 这一句要比中段任何输入都更安静、更具体
- 不要像填 KPI
- 候选可以来自本局活动日志，但只作提示，不可喧宾夺主

### Stage G：黑屏静默

**承载目标**

- 让玩家短暂承受“已经结束了”

**交互形式**

- 24:00
- 纯黑
- 绝对静音
- 5 秒左右不解释

**设计约束**

- 这段不能被按钮和加载提示破坏
- 这是整条旅程里最重要的留白之一

### Stage H：系统报告 + 讣告 + 墓志铭校对

**承载目标**

- 先用客观数字反照玩家行为
- 再用第三人称讣告给出一种“被世界转述”的感觉
- 最后用墓志铭完成最短的定稿

**顺序**

1. `最终系统报告`
2. `讣告`
3. `墓志铭校对`

**系统报告**

- 用数字墓志铭形式展示
- 不评价，只陈列

**讣告**

- 复用现有 `eulogy` 能力
- 但表现上不叫“悼词”，而像正式归档文本
- 不需要让用户点“下一步”太多次

**墓志铭校对**

- 不是“选择题”，而是“留档前核对刻字”
- 提供三条候选
- 允许用户修改一次

**删改要求**

- 取消独立“遗言一句”
- 压缩当前 `Funeral` 的多步骤表单感

### Stage I：真相揭示与回到现实

**承载目标**

- 揭示玩家没有死
- 但不抹除前面的重量
- 把体验返还给现实生活

**交互形式**

- 不用夸张反转
- 先落几句安静的解释
- 再给最后一个问题：
  - “明天醒来之后，你第一件想做得不一样的事是什么？”

**设计约束**

- 这句话与 `FinalWish` 不同
- `FinalWish` 是死亡模拟中的临终登记
- 这里是现实返还后的清醒动作

**分享卡**

- 真相揭示后静默保存
- 可让玩家自行选择是否查看或复制
- 不应在高潮点强行弹出分享冲动

## Dialogic Integration Plan

### Recommended Timelines

- `timeline_intro_boot`
- `timeline_intro_archive`
- `timeline_intro_release`
- `timeline_closure_lockdown`
- `timeline_closure_last_wish`
- `timeline_closure_obituary`
- `timeline_closure_truth`
- `timeline_closure_return`

### Recommended Variable Set

| 变量 | 用途 |
|------|------|
| `archive_work_answer` | 档案处第一问原话 |
| `archive_relation_answer` | 档案处第二问原话 |
| `archive_escape_answer` | 档案处第三问原话 |
| `archive_annotation` | 死神批注 |
| `archive_summary` | 中段叙事锚点摘要 |
| `death_cause_text` | 死因全文 |
| `closure_wish_candidate_1..3` | 临终遗愿候选 |
| `final_wish` | 最终登记遗愿 |
| `obituary_text` | 终章讣告 |
| `epitaph_1..3` | 墓志铭候选 |
| `selected_epitaph` | 玩家最终确认的墓志铭 |
| `return_to_reality_line` | 真相揭示后的最后一句 |

### Bridge Responsibilities

Dialogic 与现有 C# 系统之间需要一层清晰桥接。桥接职责包括：

- 写入 `SoulProfile`
- 触发死因生成
- 触发档案批注生成
- 触发遗愿候选生成
- 触发讣告与墓志铭候选生成
- 将 Dialogic 收到的回答写入 `GameSession`
- 从 `ActivityLog` 和统计系统取值，构造最终系统报告

注意：

- 关键桥接方法应尽量小而稳定
- 不把大量业务逻辑写进 Dialogic 脚本本身
- Dialogic 负责演出，C# 负责数据与系统边界

## Phase Remapping

为了减少工程风险，建议复用现有 Phase，而不是大规模改枚举：

| 现有阶段 | 新职责 |
|----------|--------|
| `DeathRegistration` | 黑屏通知 + 档案处对话 |
| `Verdict` | 死因裁定 + 放行 |
| `LastDay` | 高自由中段 |
| `Death` | 24:00 黑屏与静默 |
| `Funeral` | 系统报告 + 讣告 + 墓志铭 |
| `Meditation` | 真相揭示 + 现实返还问题 |
| `Ending` | 静默收束、卡片保存、退出 |

这样做的好处：

- 场景切换与 Phase 枚举改动最小
- 体验却可以完全重组
- 方便逐步迁移，不必一次性推倒重来

## Prompt Strategy

### Keep

- `grim_reaper`
- `death_cause_generation`
- `eulogy`
- `epitaph_suggestions`
- `story_summary`
- `last_day_*`

### Change

- `wish_suggestions`：
  - 不再服务于序章
  - 改为服务终章弥留遗愿候选

- `eulogy`：
  - 语义上从“悼词”偏向“讣告正文”
  - 仍可复用原有结构化输出

- `meditation_reflection`：
  - 若保留 `Meditation` 场景名，其内容不再是“冥想课程感引导”
  - 应改成“真相揭示 + 回到现实”的安静旁白

### Add

- 若需要，可新增 `archive_summary` 生成提示词，用于从开头三轮回答中提炼中段叙事锚点
- 若需要，可新增 `closure_report` 生成提示词，用于把原始统计数据整理成更精炼的系统报告文本

## Logging and Data Implications

需要保证以下内容进入日志或可被终局读取：

- 档案处三轮回答
- 死神批注
- `FinalWish` 的终章登记文本
- 墓志铭最终定稿
- 真相揭示后的“回到现实的一句”

同时建议补充区分：

- `archive_dialogue`
- `closure_ritual`
- `closure_truth`

即便最终仍复用 `reaper_dialogue / choice / note` 这些现有 kind，也应在 `context` 或 `phase` 中清晰标注，以便未来检索。

## Risks / Trade-offs

- [Risk] 关键节点固定文案过多，导致多周目重复感增强。
  - Mitigation：固定骨架下保留少量动态插槽与若干备选台词组。

- [Risk] 真相揭示放在很后面，玩家可能短暂感觉“被骗”。
  - Mitigation：在序章和终章埋微弱伏笔，避免恶作剧式反转；语气必须真诚，不调侃玩家投入。

- [Risk] 墓志铭和讣告如果仍然做成多步骤交互，会重新产生 wizard 感。
  - Mitigation：把它们做成一个连续仪式页面，只保留必要选择。

- [Risk] 中段失去 `final_wish` 后，LLM 可能更分散。
  - Mitigation：引入 `archive_summary / dangling_threads` 作为新锚点。

- [Trade-off] 保留现有 Phase 会让某些场景名与新语义不完全匹配。
  - 回报是迁移成本显著降低，后续若验证成功，再考虑重命名阶段。

## Recommended Writing Principles

- 少说“人生道理”，多写“此刻的动作、犹豫、登记、删除、停顿”。
- 死神不做导师，更像见多了档案、嘴有点冷的人。
- 系统文案必须短、硬、准。
- 真相揭示必须安静，不靠大段升华堆砌。
- 让“删掉了什么、没发出去什么、拖到了最后什么”成为叙事本身。

## What Success Looks Like

如果这次重构成功，玩家在体验上应有如下感受：

- 开头不是在填资料，而是在被死神一层层套出自己是谁。
- 中段依然像生活，不像剧本考题。
- 到终章时，遗愿终于变得沉。
- 讣告和墓志铭出现时，会短暂相信自己真的失去了这一天。
- 真相揭示后，不会觉得前面被推翻，而会觉得“我被提前交还了一次时间”。

这就是本次重构真正追求的完成度。
